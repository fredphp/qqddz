# 服务端与客户端联调测试方案

## 一、WebSocket心跳修复说明

### 问题原因
- 服务端的 `ReadDeadline` 只在收到 WebSocket Pong 帧时重置
- 客户端发送的是应用层 JSON 格式的 ping 消息
- 应用层消息不会触发 PongHandler，导致连接约 27-60 秒后超时断开

### 修复内容
**文件**: `server/internal/server/client.go`

在 `ReadPump` 的消息读取循环中，收到任何消息时都重置 `ReadDeadline`：
```go
// 🔧【修复】收到任何消息都重置读取超时，保持连接活跃
_ = c.conn.SetReadDeadline(time.Now().Add(pongWait))
```

## 三、联调测试清单

### 1. WebSocket连接测试

**测试目标**: 验证连接不会在短时间内断开

**测试步骤**:
1. 打开浏览器开发者工具 → Network → WS 标签
2. 访问游戏页面，观察 WebSocket 连接
3. 保持页面打开 2-3 分钟
4. 确认连接状态仍然是打开的

**预期结果**: 连接持续保持，不会在 27-60 秒内断开

### 2. 心跳消息测试

**浏览器控制台执行**:
```javascript
// 查看心跳日志
// 每30秒应该看到发送 ping 消息

// 手动测试心跳
window.socketCtr().initSocket()
```

### 3. 竞技场弹窗测试

**测试步骤**:
1. 进入竞技场大厅
2. 点击报名按钮
3. 等待倒计时结束
4. 确认弹窗正常弹出

**预期结果**:
- 报名成功后显示倒计时
- 倒计时结束时弹窗正常弹出
- 点击弹窗可以进入比赛

### 4. 断线重连测试

**测试步骤**:
1. 进入游戏房间
2. 临时断开网络（关闭 WiFi 或切换网络）
3. 等待 5-10 秒
4. 恢复网络
5. 观察是否自动重连

**预期结果**: 网络恢复后自动重连，游戏状态正常恢复

## 四、关键配置参数

### 服务端 (server/internal/server/client.go)
```go
const (
    writeWait   = 10 * time.Second    // 写入超时
    pongWait    = 60 * time.Second    // 读取超时（等待 pong）
    pingPeriod  = (pongWait * 9) / 10 // = 54秒，ping 发送间隔
)
```

### 客户端 (client/assets/scripts/data/socket_ctr.js)
```javascript
var _heartbeatIntervalMs = 30000   // 心跳间隔（30秒）
var _heartbeatTimeoutMs = 10000    // 心跳超时时间（10秒）
```

**说明**: 客户端 30 秒发送一次心跳，服务端 60 秒超时，心跳间隔小于超时时间，保证连接不会断开。

## 六、常见问题排查

### Q1: 连接仍然断开
**排查步骤**:
1. 确认服务端代码已更新并重启
2. 检查服务端日志是否有错误
3. 确认客户端心跳正在发送（浏览器控制台）

### Q2: 弹窗不弹出
**排查步骤**:
1. 确认 WebSocket 连接正常
2. 检查是否收到了 `arena_match_start` 消息
3. 查看服务端日志中的竞技场状态

### Q3: 重连失败
**排查步骤**:
1. 确认 reconnect_token 正确保存
2. 检查服务端会话是否还有效
3. 查看重连请求的响应
